**Blog artículo #27** — GraphQL vs REST vs tRPC para APIs en SaaS modernos 2026

### URL
`/blog/graphql-vs-rest-vs-trpc-saas-2026`

### Content
- **67% stat**: equipos SaaS pierden >8h/semana por fricción frontend-backend con APIs mal definidas
- **3 tech overviews**: REST (recursos HTTP, universalidad), GraphQL (cliente controla datos, flexibilidad), tRPC (type-safety end-to-end, monorepos TS)
- **Tabla comparativa 8 dimensiones**: type-safety, curva aprendizaje, over/under-fetching, caché HTTP, API pública, clientes no-TS, velocidad desarrollo, ecosistema
- **Ejemplos de código reales**: REST Express, GraphQL query con fields específicos, tRPC router + procedimientos Zod
- **6 errores críticos** (mal vs bien):
  1. REST sin tipos: usar codegen o tRPC en su lugar
  2. N+1 en GraphQL sin DataLoader: batch con DataLoader
  3. tRPC para API pública consumida por no-TS: REST obligatorio
  4. Exponer todo el CRUD REST: endpoints orientados a casos de uso
  5. GraphQL sin auth a nivel de campo: graphql-shield/directivas
  6. Mezclar tRPC y REST sin estrategia: regla explícita en CONTRIBUTING.md
- **Caso real**: migración REST → tRPC en SaaS B2B (47 endpoints, 8h/sem fricción, 12 bugs/mes) → -73% fricción, -100% bugs tipo, +31% velocidad features
- **Árbol de decisión 5 preguntas**: ¿API pública? → ¿TypeScript 100%? → ¿Monorepo? → ¿Datos relacionados complejos? → ¿Experiencia GraphQL?
- **Regla de oro**: nuevo SaaS TS → tRPC; API pública → REST + OpenAPI; datos muy complejos → GraphQL
